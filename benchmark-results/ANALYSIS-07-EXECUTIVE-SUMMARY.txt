❌ Memory allocation is NOT the primary cause of the 456ns hot path regression.

Evidence:
  ✓ Fast path (disabled): 0 bytes allocated → 0 ns overhead
  ✓ Fast path (no subscribers): 0 bytes allocated → 0 ns overhead
  ⚠️  Async delivery: ~72 bytes/op → ~10 ns overhead (2% of regression)
  ✗ Remaining unaccounted: ~446 ns (98% of regression)

Root Cause Hypothesis:
  → Branch prediction failure: ~200-300 ns (44-66%)
  → Volatile reads: ~10-20 ns (2-4%)
  → Virtual call overhead: ~5-10 ns (1-2%)
  → Unexplained: ~120-230 ns

┌──────────────────────────────────────────────────────────────────────────────┐
│ ESCAPE ANALYSIS                                                               │
└──────────────────────────────────────────────────────────────────────────────┘

✓ CAN be stack-allocated (zero GC pressure):
  eventBus.publish(new ProcessCreated(now, "proc-1", "Proc"));
  JIT proves event doesn't escape → stack allocation

✗ MUST be heap-allocated:
  FrameworkEvent event = new ProcessCreated(now, "proc-1", "Proc");
  this.lastEvent = event;  // Escapes to field
  eventBus.publish(event);

Effectiveness:
  Non-escaping event: ~5-10 ns (vs. ~30-50 ns with allocation)
  Heap-allocated event: ~30-50 ns (allocation + initialization)

  Root cause is likely branch prediction failure due to @Param alternation

┌──────────────────────────────────────────────────────────────────────────────┐
│ RECOMMENDATIONS                                                               │
└──────────────────────────────────────────────────────────────────────────────┘

❌ DON'T optimize (low priority, minimal benefit):
  • Allocation optimization (not the bottleneck)
  • Object pooling (high complexity, bug-prone)
  • Stack allocation (already works for non-escaping events)

✅ DO optimize (high priority, high impact):
  1. Branch prediction optimization:
     • Separate benchmarks for enabled/disabled states
     • Eliminate @Param alternation
     • Expected improvement: -200 to -300 ns

  2. Compile-time elimination (Agent 4):
     • Remove FrameworkEventBus calls entirely when disabled
     • Remove branch checks
     • Remove volatile reads
     • Expected improvement: -456 ns (100% of regression)

  3. CPU profiling (Agent 8):
     • Identify remaining ~120-230 ns
     • Use JFR/async-profiler
     • Target specific hotspots
