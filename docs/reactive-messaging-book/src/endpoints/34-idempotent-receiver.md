# 34. Idempotent Receiver

> *"An Idempotent Receiver handles duplicate messages gracefully — processing the same message twice has the same effect as processing it once."*

## Intent

In any messaging system, messages can be delivered more than once due to retries, network errors, or at-least-once delivery guarantees. An Idempotent Receiver detects and silently discards duplicate messages, ensuring that downstream state changes are applied exactly once. In JOTP, this is implemented by storing seen message IDs in the `Proc` state.

## OTP Analogy

In Erlang, idempotency is typically enforced by tagging each message with a unique reference and tracking seen refs in ETS or process state:

```erlang
handle_cast({process, MsgId, Payload}, State = #state{seen = Seen}) ->
    case sets:is_element(MsgId, Seen) of
        true  ->
            % Duplicate — ignore
            {noreply, State};
        false ->
            NewSeen = sets:add_element(MsgId, Seen),
            do_process(Payload),
            {noreply, State#state{seen = NewSeen}}
    end.
```

JOTP encodes this directly in `Proc` state using a `HashSet<UUID>` of seen message IDs.

## JOTP Implementation

The `Proc<IdempotentState, DeduplicatedMsg>` handler checks each incoming message's ID against a `HashSet<UUID>`. If the ID is already present, the message is dropped and state is unchanged. If new, the message is processed and the ID is added to the set.

```
At-least-once source
    │  msg(id=UUID-1)
    │  msg(id=UUID-1)  ← duplicate
    │  msg(id=UUID-2)
    ▼
Proc<IdempotentState, DeduplicatedMsg>
    │  seen = {UUID-1, UUID-2}
    │  handler: UUID-1 → process; UUID-1 again → skip; UUID-2 → process
    ▼
Exactly-once processing
```

Key design points:
- The `messageId` `UUID` is generated by the producer at-most-once and must not change across retries.
- The `seen` set grows indefinitely unless you implement a bounded window (LRU eviction, TTL, or time-bucketed sets).
- The entire handler is deterministic and pure — `(S, M) -> S`; no external mutation until the message is confirmed as new.
- For high-throughput scenarios, use a Bloom filter as a probabilistic first pass, then check exact ID only on positive.

## API Reference

| Usage | Description |
|-------|-------------|
| `record DeduplicatedMsg(UUID messageId, …)` | All messages carry a unique ID |
| `state.seen().contains(msg.messageId())` | Duplicate check |
| `state.withSeen(newSeen)` | Return updated state with ID added |
| `HashSet<UUID>` in `Proc` state | Seen-ID deduplication store |

## Code Example

```java
import org.acme.Proc;
import java.util.HashSet;
import java.util.Set;

// --- Message type with embedded ID ---
record PaymentRequest(
    UUID   messageId,   // unique per logical message, stable across retries
    String paymentId,
    double amount,
    String currency
) {}

// --- Idempotent state ---
record IdempotentState(
    Set<UUID>       seen,       // IDs of already-processed messages
    int             processed,  // count of unique messages handled
    int             duplicates  // count of dropped duplicates
) {
    static IdempotentState empty() {
        return new IdempotentState(new HashSet<>(), 0, 0);
    }

    IdempotentState markSeen(UUID id) {
        var newSeen = new HashSet<>(seen);
        newSeen.add(id);
        return new IdempotentState(newSeen, processed + 1, duplicates);
    }

    IdempotentState countDuplicate() {
        return new IdempotentState(seen, processed, duplicates + 1);
    }
}

// --- Idempotent payment processor ---
public class IdempotentReceiver {

    static IdempotentState handle(IdempotentState state, PaymentRequest req) {
        if (state.seen().contains(req.messageId())) {
            // Duplicate — silently drop, update counter only
            System.out.printf("[DEDUP] Skipping duplicate messageId=%s paymentId=%s%n",
                req.messageId().toString().substring(0, 8), req.paymentId());
            return state.countDuplicate();
        }

        // New message — process and record the ID
        System.out.printf("[PROCESS] paymentId=%s amount=%.2f %s%n",
            req.paymentId(), req.amount(), req.currency());
        chargePayment(req.paymentId(), req.amount());

        return state.markSeen(req.messageId());
    }

    private static void chargePayment(String id, double amount) {
        // stripe.charge(id, amount);
    }

    public static void main(String[] args) throws Exception {
        var proc = Proc.of(IdempotentState.empty(), IdempotentReceiver::handle);

        // Simulate at-least-once delivery: PAY-001's message is retried twice
        var msgId1 = UUID.randomUUID();
        var msgId2 = UUID.randomUUID();

        proc.tell(new PaymentRequest(msgId1, "PAY-001", 99.99, "USD"));
        proc.tell(new PaymentRequest(msgId1, "PAY-001", 99.99, "USD")); // duplicate → skipped
        proc.tell(new PaymentRequest(msgId2, "PAY-002", 49.99, "EUR"));
        proc.tell(new PaymentRequest(msgId1, "PAY-001", 99.99, "USD")); // duplicate → skipped

        var state = proc.ask(new PaymentRequest(UUID.randomUUID(), "NOOP", 0, "USD"))
                        .get(2, TimeUnit.SECONDS);

        System.out.printf("Processed unique: %d, Duplicates dropped: %d%n",
            state.processed(), state.duplicates());  // 2 processed, 2 dropped
    }
}
```

## Bounded Deduplication Window

```java
// For long-running services, prevent unbounded set growth:
// Use a time-bucketed approach — only deduplicate within a sliding window.

record BoundedIdempotentState(
    Deque<Map.Entry<UUID, Instant>> window,  // ordered by arrival time
    Duration                        ttl,     // how long to remember IDs
    int                             processed
) {
    BoundedIdempotentState addAndEvict(UUID id) {
        var now  = Instant.now();
        var copy = new ArrayDeque<>(window);
        copy.addLast(Map.entry(id, now));
        // Evict entries older than TTL
        copy.removeIf(e -> e.getValue().isBefore(now.minus(ttl)));
        return new BoundedIdempotentState(copy, ttl, processed + 1);
    }

    boolean hasSeen(UUID id) {
        return window.stream().anyMatch(e -> e.getKey().equals(id));
    }
}
```

## Test Pattern

```java
import org.junit.jupiter.api.Test;
import org.assertj.core.api.WithAssertions;
import java.util.concurrent.TimeUnit;

class IdempotentReceiverTest implements WithAssertions {

    Proc<IdempotentState, PaymentRequest> freshProc() {
        return Proc.of(IdempotentState.empty(), IdempotentReceiver::handle);
    }

    @Test
    void uniqueMessagesAreProcessed() throws Exception {
        var proc = freshProc();
        proc.tell(new PaymentRequest(UUID.randomUUID(), "PAY-T1", 10.0, "USD"));
        proc.tell(new PaymentRequest(UUID.randomUUID(), "PAY-T2", 20.0, "USD"));

        var state = proc.ask(new PaymentRequest(UUID.randomUUID(), "NOOP", 0, "USD"))
                        .get(2, TimeUnit.SECONDS);
        assertThat(state.processed()).isEqualTo(2);
        assertThat(state.duplicates()).isEqualTo(0);
    }

    @Test
    void duplicateMessageIsDropped() throws Exception {
        var proc  = freshProc();
        var msgId = UUID.randomUUID();

        proc.tell(new PaymentRequest(msgId, "PAY-T3", 5.0, "USD"));
        proc.tell(new PaymentRequest(msgId, "PAY-T3", 5.0, "USD")); // duplicate

        var state = proc.ask(new PaymentRequest(UUID.randomUUID(), "NOOP", 0, "USD"))
                        .get(2, TimeUnit.SECONDS);
        assertThat(state.processed()).isEqualTo(1);
        assertThat(state.duplicates()).isEqualTo(1);
    }

    @Test
    void seenSetGrowsWithEachUniqueMessage() throws Exception {
        var proc = freshProc();
        for (int i = 0; i < 5; i++)
            proc.tell(new PaymentRequest(UUID.randomUUID(), "PAY-" + i, i, "USD"));

        var state = proc.ask(new PaymentRequest(UUID.randomUUID(), "NOOP", 0, "USD"))
                        .get(2, TimeUnit.SECONDS);
        assertThat(state.seen()).hasSize(5);
    }
}
```

## Caveats & Trade-offs

**Use when:**
- The message source has at-least-once delivery semantics (most real-world brokers).
- Processing a duplicate message would cause double-charges, duplicate records, or other corruption.
- Message IDs are stable across producer retries.

**Avoid when:**
- Messages do not carry stable unique IDs — you cannot deduplicate without an ID.
- The deduplication window is unbounded and the service is long-running — implement TTL-based eviction.
- Idempotency is enforced at the database level (upsert/INSERT OR IGNORE) — application-level dedup may be redundant.
