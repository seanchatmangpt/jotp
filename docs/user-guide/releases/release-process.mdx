# Release Process

This document describes the complete process for creating, testing, and publishing JOTP releases.

## Overview

JOTP follows a structured release process to ensure quality, stability, and security of all published artifacts.

**Release Cadence:**
- **Major releases** (1.0.0, 2.0.0): Breakthrough features, potentially breaking changes - Annually
- **Minor releases** (1.1.0, 1.2.0): New features, backward compatible - Quarterly
- **Patch releases** (1.0.1, 1.0.2): Bug fixes, security patches - As needed

**Release Lifecycle:**
1. **Planning** → Feature freeze, roadmap finalization
2. **Development** → Feature implementation, testing
3. **Release Candidate** → Beta testing, validation
4. **General Availability** → Production release
5. **Maintenance** → Bug fixes, security patches

## Pre-Release Checklist

### 1. Feature Freeze

**Timeline:** 2 weeks before release

**Actions:**
- [ ] All planned features merged to main branch
- [ ] No new features accepted for this release
- [ ] Documentation updates completed
- [ ] Migration guides written (if breaking changes)
- [ ] Changelog updated with all changes

**Validation:**
```bash
# Verify branch is clean
git status

# Check all commits since last release
git log v1.0.0..HEAD --oneline

# Verify no outstanding PRs
gh pr list --state open --base main
```

### 2. Quality Gates

**Timeline:** 1 week before release

**Code Quality:**
- [ ] All tests passing: `mvnd verify`
- [ ] Code coverage ≥80%: `mvnd jacoco:report`
- [ ] No Spotless violations: `mvnd spotless:check`
- [ ] Javadoc validation: `mvnd javadoc:javadoc`
- [ ] ArchUnit rules passing: `mvnd test -Dtest=ArchTest`

**Performance:**
- [ ] Benchmarks run: `mvnd verify -Pbenchmark`
- [ ] No regressions vs. baseline
- [ ] Memory leaks detected (if any)

**Security:**
- [ ] Dependency vulnerability scan: `mvnd org.owasp:dependency-check-maven:check`
- [ ] Code security review completed
- [ ] Security advisories drafted (if applicable)

### 3. Documentation

**Timeline:** 3 days before release

**Required:**
- [ ] Changelog updated with all changes
- [ ] Migration guide for breaking changes
- [ ] API documentation updated (Javadoc)
- [ ] User guide examples tested
- [ ] Release notes drafted

**Optional:**
- [ ] Blog post drafted
- [ ] Conference talk submitted (for major releases)
- [ ] Video demo recorded

## Release Process

### Step 1: Create Release Branch

```bash
# From main branch
git checkout main
git pull origin main

# Create release branch
git checkout -b release/1.0.0

# Update version in pom.xml
# <version>1.0.0</version>

# Commit version change
git commit -am "chore: Bump version to 1.0.0"
```

### Step 2: Final Testing

**Run Full Test Suite:**
```bash
# Clean build
mvnd clean

# Full verification (unit + integration + dogfood)
mvnd verify -Ddogfood

# Run benchmarks
mvnd verify -Pbenchmark

# Run stress tests
mvnd test -Dtest=*StressTest
```

**Manual Testing:**
- [ ] Run chaos demo: `mvnd exec:java -Dexec.mainClass="io.github.seanchatmangpt.jotp.examples.ChaosDemo"`
- [ ] Run Spring Boot example
- [ ] Verify documentation examples compile and run
- [ ] Test migration guide examples (if applicable)

### Step 3: Create Git Tag

```bash
# Ensure everything is committed
git status

# Create annotated tag
git tag -a v1.0.0 -m "Release JOTP 1.0.0 - Java 26 Launch

- All 15 OTP primitives complete
- Production-ready with 100+ tests
- Java 26 virtual threads support
- Comprehensive documentation
"

# Push tag to remote
git push origin v1.0.0
```

### Step 4: Build Release Artifacts

```bash
# Clean build
mvnd clean

# Package with sources and Javadoc
mvnd package

# Create release bundle (optional)
mvnd assembly:single
```

**Artifacts Generated:**
- `jotp-1.0.0.jar` - Main artifact
- `jotp-1.0.0-sources.jar` - Source code
- `jotp-1.0.0-javadoc.jar` - API documentation
- `jotp-1.0.0.pom` - Maven POM

### Step 5: Publish to Maven Central

**Prerequisites:**
- [ ] OSSRH account configured
- [ ] GPG key configured in `~/.m2/settings.xml`
- [ ] Maven Central token configured

**Publish Snapshot (for testing):**
```bash
# Deploy to snapshot repository
mvnd deploy

# Verify at: https://s01.oss.sonatype.org/content/repositories/snapshots/
```

**Publish Release:**
```bash
# Deploy to staging repository
mvnd deploy -P release

# Login to OSSRH: https://s01.oss.sonatype.org/
# Navigate to Staging Repositories
# Select repository and click "Close"
# Wait for validation (2-5 minutes)
# Click "Release" to publish to Maven Central

# Verify at: https://central.sonatype.com/artifact/io.github.seanchatmangpt/jotp
```

### Step 6: Create GitHub Release

```bash
# Using GitHub CLI
gh release create v1.0.0 \
  --title "JOTP 1.0.0 - Java 26 Launch" \
  --notes "Release notes here..." \
  --draft \
  jotp-1.0.0.jar \
  jotp-1.0.0-sources.jar \
  jotp-1.0.0-javadoc.jar
```

**Release Notes Template:**
```markdown
## JOTP 1.0.0 - Java 26 Launch

### 🎉 Major Features

- **All 15 OTP Primitives**: Complete Erlang/OTP implementation in Java 26
- **Production-Ready**: 100+ tests, CI/CD, comprehensive documentation
- **Java 26**: Virtual threads, sealed types, pattern matching
- **Enterprise Grade**: Supervision trees, fault tolerance, self-healing

### 📊 What's New

See [Changelog](./changelog.mdx) for complete details.

### 🚀 Getting Started

```xml
<dependency>
    <groupId>io.github.seanchatmangpt</groupId>
    <artifactId>jotp</artifactId>
    <version>1.0.0</version>
</dependency>
```

### 📖 Documentation

- [Quick Start](https://github.com/seanchatmangpt/jotp#quick-start)
- [User Guide](https://github.com/seanchatmangpt/jotp/tree/main/docs/user-guide)
- [API Reference](https://github.com/seanchatmangpt/jotp/tree/main/docs/reference)

### 🐛 Bug Reports

[GitHub Issues](https://github.com/seanchatmangpt/jotp/issues)

### ⚠️ Breaking Changes

See [Migration Guide](./migration-guides.mdx)

### 🙏 Acknowledgments

Thank you to all contributors!
```

### Step 7: Post-Release Tasks

**Immediate (Day 0):**
- [ ] Publish release to GitHub (remove draft status)
- [ ] Announce on mailing list/Discord
- [ ] Tweet announcement
- [ ] Update website with new version

**Week 1:**
- [ ] Monitor GitHub Issues for bug reports
- [ ] Respond to StackOverflow questions
- [ ] Gather user feedback
- [ ] Plan patch releases if critical bugs found

**Month 1:**
- [ ] Analyze usage metrics
- [ ] Collect performance benchmarks from users
- [ ] Plan next minor/major release
- [ ] Write blog post on real-world usage

## Release Verification

### Smoke Testing

After release, verify:

```bash
# Create test project
mkdir jotp-test && cd jotp-test

# Create pom.xml with new version
cat > pom.xml <<'EOF'
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>test</groupId>
    <artifactId>jotp-test</artifactId>
    <version>1.0.0</version>
    <dependencies>
        <dependency>
            <groupId>io.github.seanchatmangpt</groupId>
            <artifactId>jotp</artifactId>
            <version>1.0.0</version>
        </dependency>
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.12.2</version>
            <scope>test</scope>
        </dependency>
    </dependencies>
</project>
EOF

# Test dependency resolution
mvnd dependency:tree

# Create simple test
mkdir -p src/test/java/test
cat > src/test/java/test/SanityTest.java <<'EOF'
package test;

import io.github.seanchatmangpt.jotp.Proc;
import org.junit.jupiter.api.Test;

class SanityTest {
    @Test
    void testProcSpawn() {
        var proc = Proc.spawn(0, (state, msg) -> state);
        proc.tell("hello");
        proc.stop();
    }
}
EOF

# Run test
mvnd test
```

### Maven Central Verification

```bash
# Wait 10-20 minutes for sync
# Then verify:

curl -I https://repo1.maven.org/maven2/io/github/seanchatmangpt/jotp/1.0.0/jotp-1.0.0.pom

# Should return HTTP 200
```

## Rollback Procedure

If critical issues are found:

### Step 1: Immediate Actions

```bash
# Yank release from Maven Central (requires OSSRH access)
# Login to OSSRH and drop the release

# Mark GitHub release as draft
gh release edit v1.0.0 --draft

# Announce rollback on communication channels
```

### Step 2: Fix Issues

```bash
# Create hotfix branch
git checkout -b hotfix/v1.0.1

# Fix issues
# Update version to 1.0.1
git commit -am "fix: Critical issue resolved"

# Create patch release
git tag -a v1.0.1 -m "Hotfix release"
git push origin v1.0.1
```

### Step 3: Republish

```bash
# Deploy hotfix
mvnd deploy -P release

# Create new GitHub release
gh release create v1.0.1 --notes "Hotfix for critical issue"
```
