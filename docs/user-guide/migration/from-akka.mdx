# Migrating from Akka to JOTP

A comprehensive guide for Akka developers moving to JOTP (Java 26 implementation of OTP primitives).

## Overview

JOTP brings Erlang/OTP's battle-tested concurrency patterns to the JVM using Java 26's virtual threads. While Akka was inspired by Erlang/OTP, JOTP provides a more direct mapping to OTP primitives with Java 26's modern language features.

**Key Differences:**
- **Virtual Threads:** JOTP uses Java 26 virtual threads instead of Akka's custom dispatcher
- **Typed Messages:** JOTP enforces type safety with sealed interfaces
- **No Configuration:** JOTP uses Java code instead of HOCON configuration
- **OTP Parity:** JOTP implements all 15 OTP primitives with full feature parity

## Concept Mapping

### Core Primitives

| Akka | JOTP | Notes |
|------|------|-------|
| `ActorRef<T>` | `Proc<S,M>` or `ProcRef<S,M>` | Typed handles with state and message types |
| `Actor` | `Proc<S,M>` handler function | Functional style vs object-oriented |
| `ActorSystem` | No equivalent | No central coordinator needed |
| `Props` | Constructor or `Proc.spawn()` | No configuration objects |
| `Inbox` | `ask()` pattern | Built-in request-reply |
| `Dispatcher` | ForkJoinPool (default) | Virtual threads use carrier threads |
| `Mailbox` | `LinkedTransferQueue` | Lock-free MPMC queue |
| `tell(!)` | `tell()` | Fire-and-forget messaging |
| `ask(?)` | `ask()` | Request-reply with timeout |

### Supervision

| Akka | JOTP | Notes |
|------|------|-------|
| `supervise()` | `supervisor.supervise()` | Direct API, no props |
| `OneForOneStrategy` | `Strategy.ONE_FOR_ONE` | Same semantics |
| `AllForOneStrategy` | `Strategy.ONE_FOR_ALL` | Different name, same behavior |
| `SupervisorStrategy` | `Strategy` enum | Simplified model |
| `childSpec()` | `ChildSpec` record | Explicit configuration |
| `restart()` | Automatic via Supervisor | No manual restart needed |

### Patterns

| Akka Pattern | JOTP Equivalent | Notes |
|--------------|-----------------|-------|
| `Actor` subclass | `Proc<S,M>` with handler | Functional vs OO |
| `AbstractBehavior` | Handler function | Pattern matching instead of methods |
| `ReceiveBuilder` | `switch` expressions | Compiler-verified exhaustiveness |
| `Stash` | `Action.postpone()` | StateMachine feature |
| `FSM` | `StateMachine<S,E,D>` | Full OTP gen_statem parity |
| `Persistence` | Not in JOTP 1.0 | Use event sourcing separately |
| `Cluster` | Not in JOTP 1.0 | Use distributed systems tools |
| `Stream` | Not in JOTP | Use Reactive Streams |

## Actor Migration

### Basic Actor

<Tabs items={['Akka', 'JOTP']}>
  <Tab>
    ```java
    public class CounterActor extends AbstractBehavior<CounterCommand> {
        public static class Increment implements CounterCommand {}
        public static class GetCount implements CounterCommand {}

        private int count = 0;

        public static Behavior<CounterCommand> create() {
            return Behaviors.setup(CounterActor::new);
        }

        private CounterActor(ActorContext<CounterCommand> context) {
            super(context);
        }

        @Override
        public Receive<CounterCommand> createReceive() {
            return newReceiveBuilder()
                .onMessage(Increment.class, msg -> {
                    count++;
                    return this;
                })
                .onMessage(GetCount.class, msg -> {
                    msg.getReplyTo().tell(count);
                    return this;
                })
                .build();
        }
    }

    // Usage
    var system = ActorSystem.create(CounterActor.create(), "counter");
    var counter = system.systemActorOf(CounterActor.create(), "counter");
    counter.tell(new Increment());
    ```
  </Tab>
  <Tab>
    ```java
    // Message protocol (sealed interface)
    sealed interface CounterMsg permits Increment, GetCount {}
    record Increment() implements CounterMsg {}
    record GetCount() implements CounterMsg {}

    // State (immutable record)
    record CounterState(int count) {}

    // Process creation
    var counter = Proc.spawn(
        new CounterState(0),
        (state, msg) -> switch (msg) {
            case Increment _ -> new CounterState(state.count() + 1);
            case GetCount _ -> state;  // ask() returns state
        }
    );

    // Usage
    counter.tell(new Increment());
    int count = counter.ask(new GetCount()).get(1, TimeUnit.SECONDS).count();
    ```
  </Tab>
</Tabs>

**Key Differences:**
1. **No ActorSystem:** Processes are created directly
2. **Functional style:** Handler function instead of class
3. **No Behaviors:** Simple state transformation function
4. **Explicit state:** Immutable record vs mutable field

### Typed Actor Context

<Tabs items={['Akka', 'JOTP']}>
  <Tab>
    ```java
    public class ContextActor extends AbstractBehavior<Command> {
        private final ActorContext<Command> context;

        private ContextActor(ActorContext<Command> context) {
            this.context = context;
        }

        @Override
        public Receive<Command> createReceive() {
            return newReceiveBuilder()
                .onMessage(SpawnChild.class, msg -> {
                    var child = context.spawn(ChildActor.create(), "child-" + msg.id());
                    msg.getReplyTo().tell(child);
                    return this;
                })
                .onMessage(LogSelf.class, msg -> {
                    System.out.println(context.getSelf());
                    System.out.println(context.getChildren());
                    return this;
                 })
                .build();
        }
    }
    ```
  </Tab>
  <Tab>
    ```java
    // JOTP doesn't have ActorContext
    // Child spawning is done via Supervisor or directly
    sealed interface Command permits SpawnChild, LogSelf {}
    record SpawnChild(String id, CompletableFuture<ProcRef<ChildState, ChildMsg>> replyTo) implements Command {}
    record LogSelf() implements Command {}

    var parent = Proc.spawn(
        new ParentState(Map.of()),
        (state, msg) -> switch (msg) {
            case SpawnChild(var id, var replyTo) -> {
                var child = Proc.spawn(new ChildState(), childHandler);
                var childRef = new ProcRef<>(child);
                replyTo.complete(childRef);
                yield new ParentState(merge(state.children(), Map.of(id, childRef)));
            }
            case LogSelf _ -> {
                System.out.println("No self-reference needed in JOTP");
                System.out.println("Children tracked in state: " + state.children());
                yield state;
            }
        }
    );
    ```
  </Tab>
</Tabs>

**Key Differences:**
1. **No built-in hierarchy:** Parent tracks children manually
2. **State-based:** Child references in state, not context
3. **Explicit lifecycle:** Manual spawning and tracking

## State Machine Migration

### Akka FSM

<Tabs items={['Akka FSM', 'JOTP StateMachine']}>
  <Tab>
    ```java
    public class Lock extends AbstractFSM<LockState, LockData> {
        {
            startWith(Locked, new LockData("1234", ""));

            when(Locked,
                matchEvent(
                    Button.class,
                    (button, data) -> {
                        var newInput = data.input() + button.digit();
                        if (newInput.equals(data.correctCode())) {
                            return goTo(Open)
                                .using(new LockData(data.correctCode(), ""))
                                .forMax(Duration.ofSeconds(10));
                        } else if (newInput.length() >= 4) {
                            return stay().using(new LockData(data.correctCode(), ""));
                        } else {
                            return stay().using(new LockData(data.correctCode(), newInput));
                        }
                    }
                )
            );

            when(Open,
                matchEvent(
                    StateTimeout.class,
                    (timeout, data) -> goTo(Locked)
                )
                .matchEvent(
                    Lock.class,
                    (lock, data) -> goTo(Locked)
                )
            );

            initialize();
        }
    }
    ```
  </Tab>
  <Tab>
    ```java
    sealed interface LockState permits Locked, Open {}
    record Locked() implements LockState {}
    record Open() implements LockState {}

    sealed interface LockEvent permits Button, Lock {}
    record Button(char digit) implements LockEvent {}
    record Lock() implements LockEvent {}

    record LockData(String correctCode, String input) {}

    var lockMachine = StateMachine.of(
        new Locked(),
        new LockData("1234", ""),
        (state, event, data) -> switch (state) {
            case Locked _ -> switch (event) {
                case SMEvent.User(Button(var digit)) -> {
                    var newInput = data.input() + digit;
                    if (newInput.equals(data.correctCode())) {
                        yield Transition.nextState(
                            new Open(),
                            new LockData(data.correctCode(), ""),
                            Action.stateTimeout(10_000, new Lock())
                        );
                    } else if (newInput.length() >= 4) {
                        yield Transition.keepState(new LockData(data.correctCode(), ""));
                    } else {
                        yield Transition.keepState(new LockData(data.correctCode(), newInput));
                    }
                }
                case SMEvent.StateTimeout(var _) ->
                    yield Transition.keepState(new LockData(data.correctCode(), ""));
                default -> Transition.keepState(data);
            };

            case Open _ -> switch (event) {
                case SMEvent.User(Lock _), SMEvent.StateTimeout(var _) ->
                    yield Transition.nextState(new Locked(), data);
                default ->
                    yield Transition.keepState(data, Action.postpone());
            };
        }
    );
    ```
  </Tab>
</Tabs>

**Key Differences:**
1. **Explicit state/data:** Separation of concerns
2. **No DSL:** Pure Java with pattern matching
3. **Full OTP parity:** State timeouts, postpone, etc.
4. **Type-safe:** Compiler verifies all cases

## Message Patterns

### Become/Unbecome

<Tabs items={['Akka', 'JOTP']}>
  <Tab>
    ```java
    public class StatefulActor extends AbstractBehavior<Command> {
        private Behavior<Command> active = Behaviors.receiveMessage(msg -> {
            if (msg instanceof Stop) {
                return idle;
            }
            // Handle active state
            return this;
        });

        private Behavior<Command> idle = Behaviors.receiveMessage(msg -> {
            if (msg instanceof Start) {
                return active;
            }
            // Handle idle state
            return this;
        });

        @Override
        public Receive<Command> createReceive() {
            return newReceiveBuilder().onMessage(Command.class, msg -> idle).build();
        }
    }
    ```
  </Tab>
  <Tab>
    ```java
    sealed interface ActorState permits Idle, Active {}
    record Idle() implements ActorState {}
    record Active() implements ActorState {}

    sealed interface Command permits Start, Stop, Other {}
    record Start() implements Command {}
    record Stop() implements Command {}
    record Other(String data) implements Command {}

    var actor = Proc.spawn(
        new Idle(),
        (state, msg) -> switch (state) {
            case Idle _ -> switch (msg) {
                case Start _ -> new Active();
                case Other(var data) -> {
                    // Ignore or queue
                    yield new Idle();
                }
            };
            case Active _ -> switch (msg) {
                case Stop _ -> new Idle();
                case Other(var data) -> {
                    // Process message
                    yield new Active();
                }
            };
        }
    );
    ```
  </Tab>
</Tabs>

**Key Differences:**
1. **State in data:** Not in handler function
2. **Explicit switching:** Switch on state and message
3. **Type-safe:** State is part of the process type

### Stash Pattern

<Tabs items={['Akka', 'JOTP']}>
  <Tab>
    ```java
    public class StashActor extends AbstractBehavior<Command>
        implements StashFactory<Command> {

        private final Stash<Command> stash;

        private StashActor(ActorContext<Command> context) {
            super(context);
            this.stash = StashFactory.create(context, this);
        }

        @Override
        public Receive<Command> createReceive() {
            return newReceiveBuilder()
                .onMessage(Initialize.class, msg -> {
                    stash.unstashAll();
                    return initialized();
                })
                .onMessage(Command.class, msg -> {
                    stash.stash(msg);
                    return this;
                })
                .build();
        }

        private Receive<Command> initialized() {
            return newReceiveBuilder()
                .onMessage(Command.class, msg -> {
                    // Handle normally
                    return this;
                })
                .build();
        }
    }
    ```
  </Tab>
  <Tab>
    ```java
    sealed interface State permits Uninitialized, Initialized {}
    record Uninitialized(List<Command> stashed) implements State {}
    record Initialized() implements State {}

    var actor = Proc.spawn(
        new Uninitialized(List.of()),
        (state, msg) -> switch (state) {
            case Uninitialized(var stashed) -> switch (msg) {
                case Initialize _ -> {
                    // Process all stashed messages
                    var newState = new Initialized();
                    for (var m : stashed) {
                        newState = handleInInitialized(newState, m);
                    }
                    yield newState;
                }
                case Command _ -> {
                    // Stash message
                    yield new Uninitialized(append(stashed, msg));
                }
            };
            case Initialized _ -> handleInInitialized(state, msg);
        }
    );
    ```
  </Tab>
</Tabs>

**Key Differences:**
1. **Manual stash:** Track in state
2. **Explicit processing:** Loop through stashed messages
3. **More control:** Can implement custom stash logic

## Common Gotchas

### 1. No ActorSystem

**Problem:** Looking for a central ActorSystem.

**Solution:** Create processes directly; no central coordinator needed.

```java
// WRONG
var system = ActorSystem.create(...);

// CORRECT
var proc = Proc.spawn(state, handler);
```

### 2. No Context Self-Reference

**Problem:** Need to reference "self" in messages.

**Solution:** Use ProcRef or pass references explicitly.

```java
// WRONG
context.getSelf().tell(msg);

// CORRECT
var selfRef = new ProcRef<>(proc);
selfRef.tell(msg);
```

### 3. Message Ordering Guarantees

**Problem:** Akka guarantees message order per sender.

**Solution:** JOTP provides the same guarantee via mailbox.

```java
// Same guarantee
proc1.tell(msg1);
proc1.tell(msg2);
// proc1 processes msg1 then msg2
```

### 4. Dispatcher Performance

**Problem:** Trying to optimize dispatcher configuration.

**Solution:** Virtual threads handle scheduling efficiently.

```java
// NOT NEEDED
akka {
  actor {
    default-dispatcher {
      throughput = 5
    }
  }
}

// JVM handles this automatically
```

### 5. Remote Actors

**Problem:** Need distributed actors.

**Solution:** Use distributed systems tools.

```java
// NOT IN JOTP 1.0
context.spawn(..., "remote@host:port")

// INSTEAD: Use Kafka, gRPC, etc.
var producer = createKafkaProducer();
producer.send(new ProducerRecord<>("topic", msg));
```

## Best Practices for Migration

### 1. Start with Stateless Actors
Migrate simple actors first to learn the functional style.

### 2. Use Sealed Types Extensively
Leverage the compiler for exhaustive message handling.

```java
sealed interface Command permits A, B, C {}
record A() implements Command {}
record B() implements Command {}
record C() implements Command {}
```

### 3. Embrace Immutability
All state should be immutable records.

```java
public record UserState(
    String id,
    String name,
    List<String> roles  // Immutable list
) {}
```

### 4. Test Supervision Trees
Verify crash recovery works as expected.

```java
@Test
void testSupervisorRestarts() {
    var sup = Supervisor.create(ONE_FOR_ONE, 5, Duration.ofSeconds(60));
    var child = sup.supervise("test", state, handler);

    // Crash child
    child.proc().thread().interrupt();

    // Verify restart
    assertTrue(child.proc().isRunning());
}
```

### 5. Monitor Process Health
Use ProcSys for introspection.

```java
var stats = ProcSys.of(proc).statistics();
System.out.println("Messages: " + stats.messagesIn());
System.out.println("Queue: " + stats.queueDepth());
```

## Performance Considerations

### Virtual Thread Advantages
- **Scalability:** Millions of processes
- **Memory:** ~1 KB per process
- **Latency:** Sub-millisecond context switching

### When to Use Platform Threads
- Heavy CPU computation
- JNI calls with pinning
- Legacy integration points

## Rollback Strategy

### Phase 1: Side-by-Side Deployment
- Run Akka and JOTP versions in parallel
- Compare behavior and performance
- Gradually shift traffic to JOTP

### Phase 2: Feature Flags
- Use feature flags to switch between implementations
- Monitor error rates and latency
- Quick rollback via configuration

### Phase 3: Database Compatibility
- Ensure both versions use same database schema
- Implement data migration if needed
- Test rollback with production data
