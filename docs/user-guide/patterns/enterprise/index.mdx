## Overview

JOTP's enterprise patterns provide battle-tested resilience strategies for distributed systems. Each pattern is implemented using JOTP's core primitives, leveraging virtual threads, sealed types, and process supervision to create fault-tolerant, self-healing systems.

### Why Enterprise Patterns?

Modern distributed systems face seven fundamental challenges:

1. **Cascading Failures** - One component's failure propagates through the system
2. **Resource Exhaustion** - Unbounded queues and memory leaks
3. **Slow Downstream Services** - Threads blocked waiting for responses
4. **Multi-Tenant Contention** - Noisy neighbors affecting other tenants
5. **Distributed Transactions** - Maintaining consistency across services
6. **Health Monitoring** - Detecting and responding to degradation
7. **Event Delivery** - Reliable pub-sub messaging at scale

JOTP's enterprise patterns address each challenge using Erlang/OTP's "let it crash" philosophy adapted for Java 26.

## The 8 Enterprise Patterns

### Quick Reference Matrix

| Pattern | Primary Purpose | JOTP Primitive | State Machine | Use Case |
|---------|----------------|----------------|---------------|----------|
| **Circuit Breaker** | Prevent cascading failures | `Supervisor` + `Proc` | CLOSED → OPEN → HALF_OPEN | Downstream service protection |
| **Backpressure** | Flow control & timeout adaptation | `Proc` + `ProcRef` | HEALTHY → WARNING → CIRCUIT_OPEN | Slow consumer protection |
| **Bulkhead Isolation** | Resource containment | `Proc` + `Semaphore` | HEALTHY → DEGRADED → EXHAUSTED | Feature capacity limits |
| **Event Bus** | Reliable pub-sub messaging | `Proc` + `EventManager` | RUNNING → DEGRADED → PAUSED | Domain event broadcasting |
| **Health Check** | Service liveness monitoring | `Proc` + `ProcMonitor` | HEALTHY → DEGRADED → UNHEALTHY | Kubernetes readiness probes |
| **Multi-Tenant Supervisor** | Tenant workload isolation | `Supervisor` hierarchy | Root (ONE_FOR_ONE) → Tenant (ONE_FOR_ALL) | SaaS platform isolation |
| **Enterprise Recovery** | Retry with exponential backoff | `Proc` + `CrashRecovery` | INITIAL → BACKOFF → RETRY | Transient fault handling |
| **Distributed Saga** | Multi-step transaction coordination | `Proc` + `StateMachine` | PENDING → IN_PROGRESS → COMPLETED/COMPENSATED | Distributed transactions |

## Pattern Composition

Patterns can be combined to create sophisticated resilience strategies. Here's how they work together:

### Layer 1: Foundation Patterns
- **Circuit Breaker** - Outer protection, fails fast when downstream fails
- **Backpressure** - Middle layer, adapts timeouts based on observed latency
- **Bulkhead Isolation** - Inner layer, limits resource consumption per feature

### Layer 2: Coordination Patterns
- **Event Bus** - Decouples components with reliable event delivery
- **Distributed Saga** - Coordinates multi-step transactions with compensation
- **Multi-Tenant Supervisor** - Isolates tenant workloads in separate supervision trees

### Layer 3: Observation Patterns
- **Health Check Manager** - Monitors service health without killing it
- **Enterprise Recovery** - Retries failed operations with intelligent backoff

### Composition Example: E-Commerce Checkout

```java
// Layer 1: Protect downstream services
var paymentBreaker = CircuitBreakerPattern.create(
    CircuitBreakerConfig.of("payment-gateway")
);

var inventoryBackpressure = Backpressure.create(
    BackpressureConfig.builder("inventory-service")
        .initialTimeout(Duration.ofMillis(500))
        .maxTimeout(Duration.ofSeconds(5))
        .build()
);

var shippingBulkhead = BulkheadIsolationEnterprise.create(
    new BulkheadConfig("shipping", List.of(
        new ResourceLimit.MaxConcurrentRequests(10)
    ), Duration.ofSeconds(5))
);

// Layer 2: Coordinate transaction
var saga = DistributedSagaCoordinator.create(
    new SagaConfig("checkout", List.of(
        new SagaStep.Action<>("reserve-inventory",
            _ -> inventoryBackpressure.execute(
                timeout -> reserveInventory(), timeout)),
        new SagaStep.Compensation<>("release-inventory",
            result -> releaseInventory(result)),
        new SagaStep.Action<>("process-payment",
            _ -> paymentBreaker.execute(
                timeout -> processPayment(), timeout)),
        new SagaStep.Compensation<>("refund-payment",
            result -> refundPayment(result)),
        new SagaStep.Action<>("schedule-shipping",
            _ -> shippingBulkhead.execute(
                () -> scheduleShipping()))
    ))
);

// Layer 3: Monitor health
var healthCheck = HealthCheckManager.create(
    new HealthCheckConfig("checkout-service",
        List.of(
            new HealthCheck.Liveness("payment-gateway", 5000),
            new HealthCheck.Liveness("inventory-service", 3000)
        ), 3, Duration.ofSeconds(30))
);

// Execute with automatic compensation on failure
CompletableFuture<SagaResult> result = saga.execute();
```

## Choose Your Pattern: Decision Tree

### 1. Is a downstream service failing repeatedly?
**YES → Circuit Breaker**
```java
var breaker = CircuitBreakerPattern.create(
    CircuitBreakerConfig.of("unstable-service")
);
breaker.execute(timeout -> callDownstream(timeout), Duration.ofSeconds(2))
```

**NO → Continue to question 2**

---

### 2. Are you experiencing resource exhaustion (memory, threads, connections)?
**YES → Bulkhead Isolation**
```java
var bulkhead = BulkheadIsolationEnterprise.create(
    new BulkheadConfig("expensive-feature",
        List.of(new ResourceLimit.MaxConcurrentRequests(20)),
        Duration.ofSeconds(10))
);
bulkhead.execute(() -> expensiveOperation());
```

**NO → Continue to question 3**

---

### 3. Are requests timing out due to slow downstream services?
**YES → Backpressure**
```java
var backpressure = Backpressure.create(
    BackpressureConfig.builder("slow-service")
        .initialTimeout(Duration.ofMillis(100))
        .maxTimeout(Duration.ofSeconds(5))
        .build()
);
backpressure.execute(timeout -> callSlowService(timeout), Duration.ofMillis(100));
```

**NO → Continue to question 4**

---

### 4. Do you need to coordinate multiple steps in a transaction?
**YES → Distributed Saga**
```java
var saga = DistributedSagaCoordinator.create(
    new SagaConfig("order-fulfillment", List.of(
        new SagaStep.Action<>("validate-order", _ -> validate()),
        new SagaStep.Action<>("reserve-inventory", _ -> reserve()),
        new SagaStep.Action<>("process-payment", _ -> pay()),
        new SagaStep.Compensation<>("refund", _ -> refund())
```

### Scenario 3: Microservices Orchestration (Distributed Transaction)

**Challenge**: Order fulfillment requires coordinating inventory, payment, and shipping across services.

**Solution**: Distributed Saga + Circuit Breaker + Enterprise Recovery

```java
var orderSaga = DistributedSagaCoordinator.create(
    new SagaConfig("order-fulfillment", List.of(
        new SagaStep.Action<>("validate-payment",
            _ -> paymentBreaker.execute(
                timeout -> paymentService.validate(), timeout)),
        new SagaStep.Action<>("reserve-inventory",
            _ -> inventoryBackpressure.execute(
                timeout -> inventoryService.reserve(), timeout)),
        new SagaStep.Action<>("charge-payment",
            _ -> recovery.retry(() -> paymentService.charge())),
        new SagaStep.Action<>("schedule-shipping",
            _ -> shippingBulkhead.execute(
                () -> shippingService.schedule())),
        new SagaStep.Compensation<>("release-inventory",
            result -> inventoryService.release(result)),
        new SagaStep.Compensation<>("refund-payment",
            result -> paymentService.refund(result))
    ))
);

// Execute saga with automatic compensation on failure
CompletableFuture<SagaResult> result = orderSaga.execute();
result.thenAccept(r -> {
    if (r.status() == SagaResult.Status.COMPLETED) {
        notifyCustomer("Order shipped!");
    } else if (r.status() == SagaResult.Status.COMPENSATED) {
        notifyCustomer("Order failed, payment refunded.");
    }
});
```

**Result**: Exactly-once transaction semantics, automatic compensation on failure, no lost or orphaned resources.

## Pattern Implementation Details

Each enterprise pattern is built on JOTP's core primitives:

### Circuit Breaker Implementation

- **Primitive**: `Supervisor` with ONE_FOR_ONE strategy
- **Mechanism**: Supervisor restart limit = circuit threshold
- **States**: CLOSED → OPEN → HALF_OPEN → CLOSED
- **Fail-Fast**: Supervisor crashes when restart limit exceeded

```java
// Circuit breaker uses Supervisor's restart intensity
Supervisor supervisor = Supervisor.create(
    Supervisor.Strategy.ONE_FOR_ONE,
    3,  // maxRestarts = failure threshold
    Duration.ofSeconds(60)
);
// After 3 crashes in 60 seconds, supervisor crashes (circuit opens)
```

### Backpressure Implementation

- **Primitive**: `Proc` with sliding window state
- **Mechanism**: Timeout adaptation based on success rate
- **States**: HEALTHY → WARNING → CIRCUIT_OPEN
- **Adaptive**: Timeout doubles on failures, resets on success

```java
// Backpressure tracks success rate in sliding window
record BackpressureState(
    Deque<Boolean> resultWindow,  // true=success, false=failure
    Duration currentTimeout
);

// Timeout adapts: 100ms → 200ms → 400ms → 800ms → 1.6s → ... → 5s (max)
```

### Bulkhead Isolation Implementation

- **Primitive**: `Proc` + `Semaphore` for concurrency limiting
- **Mechanism**: Permit-based resource control
- **States**: HEALTHY → DEGRADED → EXHAUSTED
- **Non-Blocking**: `tryAcquire()` with timeout prevents queue buildup

```java
// Bulkhead uses semaphore for concurrent request limiting
Semaphore semaphore = new Semaphore(maxConcurrent);
boolean acquired = semaphore.tryAcquire(queueTimeout.getSeconds(), TimeUnit.SECONDS);
if (!acquired) {
    return Result.failure(new BulkheadException("Request timeout"));
}
```

### Event Bus Implementation

- **Primitive**: `Proc` + `EventManager` for pub-sub
- **Mechanism**: Sequential delivery to subscribers with DLQ
- **States**: RUNNING → DEGRADED → PAUSED
- **Guarantees**: At-least-once delivery with retries

```java
// Event bus delivers to all subscribers sequentially
for (SubscriberHandler subscriber : subscribers.values()) {
    try {
        subscriber.handler().accept(event);
    } catch (Exception e) {
        if (config.deadLetterQueueEnabled()) {
            deadLetterQueue.addLast(event);  // Failed event goes to DLQ
        }
    }
}
```

### Health Check Implementation

- **Primitive**: `Proc` + `ProcMonitor` (non-invasive monitoring)
- **Mechanism**: Periodic checks with consecutive failure threshold
- **States**: HEALTHY → DEGRADED → UNHEALTHY → UNREACHABLE
- **Non-Killing**: Uses `ProcMonitor`, not `ProcLink` (monitor side unaffected by crash)

```java
// Health check uses ProcMonitor (one-way notification)
ProcMonitor.monitor(targetProc, (reason) -> {
    // Target crashed, but health checker survives
    coordinator.tell(new HealthCheckMsg.CheckFailed(checkName, reason));
});
```

### Multi-Tenant Supervisor Implementation

- **Primitive**: Hierarchical `Supervisor` trees
- **Mechanism**: Root (ONE_FOR_ONE) → Tenant (ONE_FOR_ALL)
- **States**: ACTIVE → RESTARTING → SHUTDOWN
- **Isolation**: One tenant's crash doesn't affect other tenants

```java
// Multi-tenant supervisor hierarchy
Supervisor root = Supervisor.create(ONE_FOR_ONE);  // Restarts individual tenants
Supervisor tenantA = Supervisor.create(ONE_FOR_ALL);  // Restarts all tenant services
Supervisor tenantB = Supervisor.create(ONE_FOR_ALL);

root.supervise("tenant-a", tenantA);
root.supervise("tenant-b", tenantB);

// If tenant-a crashes, tenant-b is unaffected
```

### Enterprise Recovery Implementation

- **Primitive**: `Proc` + `CrashRecovery` pattern
- **Mechanism**: Fresh virtual thread per attempt with exponential backoff
- **States**: INITIAL → BACKOFF → RETRY → COMPLETED/FAILED
- **Jitter**: Randomized delay prevents thundering herd

```java
// Recovery uses CrashRecovery pattern (new thread per attempt)
Duration delay = calculateBackoff(attemptNumber);
// delay = initial * (multiplier ^ attempt) ± jitter
// Example: 100ms → 200ms → 400ms → 800ms → 1.6s → ... → 5s (max)

Thread.sleep(delay.toMillis());  // Backoff delay
return executeWithRetry(task, attemptNumber + 1);  // Recursive retry
```

### Distributed Saga Implementation

- **Primitive**: `Proc` + `StateMachine` for orchestration
- **Mechanism**: Forward actions with compensating transactions
- **States**: PENDING → IN_PROGRESS → COMPLETED/COMPENSATED
- **Compensation**: Executes compensations in reverse order on failure

```java
// Saga executes forward actions, then compensations in reverse on failure
for (int i = 0; i < steps.size(); i++) {
    try {
        outputs.put(steps.get(i).name(), execute(step));
    } catch (Exception e) {
        // Compensate in reverse order: i-1, i-2, ..., 0
        for (int j = i - 1; j >= 0; j--) {
            compensate(steps.get(j));
        }
        return SagaResult.COMPENSATED;
    }
}
```

## Advanced Topics

### Composing Multiple Patterns

Patterns can be layered for sophisticated resilience:

```java
// Layer 1: Circuit breaker (outer protection)
var breaker = CircuitBreakerPattern.create(
    CircuitBreakerConfig.of("downstream-service")
);

// Layer 2: Backpressure (timeout adaptation)
var backpressure = Backpressure.create(
    BackpressureConfig.builder("downstream-service")
        .initialTimeout(Duration.ofMillis(100))
        .maxTimeout(Duration.ofSeconds(5))
        .build()
);

// Layer 3: Bulkhead (resource isolation)
var bulkhead = BulkheadIsolationEnterprise.create(
    new BulkheadConfig("downstream-service",
        List.of(new ResourceLimit.MaxConcurrentRequests(10)),
        Duration.ofSeconds(5))
);

// Layer 4: Recovery (retry logic)
var recovery = EnterpriseRecovery.create(
    new RecoveryConfig("downstream-service",
        3, Duration.ofMillis(100), Duration.ofSeconds(5), 2.0, 0.1)
);

// Compose all layers
Result<String> result = recovery.retry(() ->
    breaker.execute(timeout ->
        backpressure.execute(bpTimeout ->
            bulkhead.execute(() ->
                callDownstreamService()
            ), bpTimeout
        ), timeout
    )
);
```

## Troubleshooting

### Circuit Breaker Issues

**Problem**: Circuit stuck in OPEN state
```java
// Check reset timeout
if (breaker.getState().status() == CircuitState.Status.OPEN) {
    long timeSinceFailure = System.currentTimeMillis() - breaker.getState().lastFailureTime();
    if (timeSinceFailure >= config.resetTimeout().toMillis()) {
        breaker.reset();  // Manual reset
    }
}
```

**Problem**: Circuit trips too frequently
```java
// Increase failure threshold and reset timeout
```
