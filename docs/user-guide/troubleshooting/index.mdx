## Overview

This comprehensive troubleshooting guide provides systematic procedures for diagnosing and resolving issues across all layers of JOTP applications, from build problems to distributed system failures.

<Callout type="info">
  **Quick Start:** Start with the [Diagnostic Checklist](/docs/user-guide/troubleshooting/diagnostic-checklist) to identify your issue category, then navigate to the specific guide for detailed solutions.
</Callout>

## Getting Started

### Step 1: Identify Your Issue Category

Use the decision tree below to identify which guide covers your issue:

```
Is it a compilation/build issue?
├─ Yes → Build Issues Guide
└─ No
   ├─ Is it about process lifecycle or messaging?
   │  ├─ Yes → Runtime Issues Guide
   │  └─ No
   │      ├─ Is it about supervision trees?
   │      │  ├─ Yes → Supervision Issues Guide
   │      │  └─ No
   │          ├─ Is it about performance (latency, memory, GC)?
   │          │  ├─ Yes → Performance Issues Guide
   │          │  └─ No
   │              ├─ Is it about testing?
   │              │  ├─ Yes → Testing Issues Guide
   │              │  └─ No
   │                  ├─ Is it about production deployment?
   │                  │  ├─ Yes → Production Issues Guide
   │                  │  └─ No
   │                      ├─ Is it about enterprise patterns?
   │                      │  ├─ Yes → Enterprise Pattern Issues Guide
   │                      │  └─ No
   │                          ├─ Is it about distributed systems?
   │                          │  ├─ Yes → Distributed Issues Guide
   │                          │  └─ No
   │                              └─ Is it about combining patterns?
   │                                 └─ Yes → Pattern Composition Issues Guide
```

### Step 2: Enable Debug Logging

```java
// Enable comprehensive JOTP logging
import io.github.seanchatmangpt.jotp.debug.DebugOptions;

DebugOptions options = DebugOptions.builder()
    .traceAll(true)
    .logMessageHandling(true)
    .logStateTransitions(true)
    .logSupervision(true)
    .build();

Proc.setGlobalDebugOptions(options);
```

### Step 3: Collect Diagnostic Information

```bash
# Thread dump
jcmd <pid> Thread.print_to_file -file=/tmp/threads.txt

# Heap information
jcmd <pid> GC.heap_info

# JOTP process summary
jcmd <pid> ProcSys.summary

# List all processes
jcmd <pid> ProcSys.listProcesses

# Virtual thread summary
jcmd <pid> Thread.vthread_summary
```

## Common Error Quick Reference

| Error | Quick Fix | Guide |
|-------|-----------|-------|
| `ClassCastException` in handler | Use sealed interfaces with exhaustive matching | [Common Errors](/docs/user-guide/troubleshooting/common-errors) |
| Process already registered | Check registry before registering | [Common Errors](/docs/user-guide/troubleshooting/common-errors) |
| `TimeoutException` in ask() | Always use timeouts with fallback | [Common Errors](/docs/user-guide/troubleshooting/common-errors) |
| Mailbox overflow | Implement backpressure or bounded mailboxes | [Runtime Issues](/docs/user-guide/troubleshooting/runtime-issues) |
| Circuit breaker stuck open | Check half-open timeout and success detection | [Enterprise Issues](/docs/user-guide/troubleshooting/enterprise-issues) |
| Children not restarting | Check restart intensity and strategy | [Supervision Issues](/docs/user-guide/troubleshooting/supervision-issues) |
| OutOfMemoryError | Capture heap dump, analyze for leaks | [Performance Issues](/docs/user-guide/troubleshooting/performance-issues) |
| Flaky test | Use Awaitility with proper waits | [Testing Issues](/docs/user-guide/troubleshooting/testing-issues) |
| Virtual thread pinning | Replace synchronized with ReentrantLock | [Runtime Issues](/docs/user-guide/troubleshooting/runtime-issues) |
| Cluster not forming | Check network connectivity and firewall | [Distributed Issues](/docs/user-guide/troubleshooting/distributed-issues) |

## Diagnostic Workflow

### Phase 1: Initial Assessment

<Callout type="info">
  **Time Estimate:** 5-10 minutes
</Callout>

1. **Identify Symptoms**
   - What are you observing?
   - When did it start?
   - Is it affecting all users or specific ones?
   - Is it intermittent or constant?

2. **Check System Health**
   ```bash
   # Check process status
   jcmd <pid> ProcSys.summary

   # Check system resources
   top -p <pid>
   free -h
   df -h

   # Check logs for errors
   grep -i error /var/log/jotp/application.log | tail -100
   ```

3. **Collect Quick Metrics**
   ```bash
   # Process count
   jcmd <pid> ProcSys.listProcesses | wc -l

   # Memory usage
   jcmd <pid> GC.heap_info

   # Thread count
   jcmd <pid> Thread.vthread_summary
   ```

### Phase 2: Categorization

<Callout type="info">
  **Time Estimate:** 5-15 minutes
</Callout>

Use the decision tree or [Diagnostic Checklist](/docs/user-guide/troubleshooting/diagnostic-checklist) to categorize your issue.

**Key Questions:**
- Is it a build/compilation issue? → [Build Issues](/docs/user-guide/troubleshooting/build-issues)
- Is it about process lifecycle? → [Runtime Issues](/docs/user-guide/troubleshooting/runtime-issues)
- Is it about supervision? → [Supervision Issues](/docs/user-guide/troubleshooting/supervision-issues)
- Is it about performance? → [Performance Issues](/docs/user-guide/troubleshooting/performance-issues)
- Is it about testing? → [Testing Issues](/docs/user-guide/troubleshooting/testing-issues)
- Is it production-related? → [Production Issues](/docs/user-guide/troubleshooting/production-issues)
- Is it about enterprise patterns? → [Enterprise Issues](/docs/user-guide/troubleshooting/enterprise-issues)
- Is it distributed systems? → [Distributed Issues](/docs/user-guide/troubleshooting/distributed-issues)
- Is it pattern composition? → [Pattern Composition Issues](/docs/user-guide/troubleshooting/pattern-composition-issues)

### Phase 3: Detailed Diagnosis

<Callout type="info">
  **Time Estimate:** 15-30 minutes
</Callout>

Follow the diagnostic steps in the relevant guide:

1. **Enable Specific Debugging**
   ```java
   // For runtime issues
   DebugOptions options = DebugOptions.builder()
       .traceAll(true)
       .logMessageHandling(true)
       .build();

   // For performance issues
   -XX:StartFlightRecording=filename=recording.jfr,duration=60s

   // For distributed issues
   -Dio.github.seanchatmangpt.jotp.distributed.level=FINEST
   ```

2. **Collect Detailed Information**
   - Thread dumps
   - Heap dumps (if memory issue)
   - JFR recordings (if performance issue)
   - Network diagnostics (if distributed issue)
   - Application logs

3. **Analyze Patterns**
   - Look for correlations in timestamps
   - Check for resource exhaustion
   - Identify bottlenecks
   - Find error patterns

### Phase 4: Solution Implementation

<Callout type="info">
  **Time Estimate:** Variable (30 minutes to several hours)
</Callout>

1. **Try Quick Fixes First**
   - Each guide has "Quick Fix" sections
   - Start with the most likely solutions
   - Test one change at a time

2. **Implement Systematic Solutions**
   - Follow the detailed solution steps
   - Make incremental changes
   - Test after each change

3. **Verify the Fix**
   ```bash
   # Check system health
   jcmd <pid> ProcSys.summary

   # Monitor metrics
   curl http://localhost:8080/metrics

   # Run relevant tests
   mvnd test -Dtest=RelevantTest
   ```

### Phase 5: Prevention

<Callout type="info">
  **Time Estimate:** Ongoing
</Callout>

1. **Add Monitoring**
   - Set up alerts for the issue
   - Create dashboards for key metrics
   - Implement automated checks

2. **Update Documentation**
   - Document the issue and solution
   - Update runbooks
   - Share with team

3. **Prevent Recurrence**
   - Add tests to catch the issue
   - Improve code review checklist
   - Update best practices guide

## Key Diagnostic Commands

### Process Inspection

```java
// Check process state
ProcInspection inspect = ProcSys.inspect(procId);
System.out.println("State: " + inspect.state());
System.out.println("Mailbox: " + inspect.mailboxSize());
System.out.println("Alive: " + inspect.isAlive());

// List all processes
Set<ProcId> all = ProcSys.allProcesses();

// Find processes by name
List<ProcRef<?, ?>> processes = ProcRegistry.getInstance().lookup("payment-service");
```

### JVM Diagnostics

```bash
# Thread dump
jcmd <pid> Thread.print_to_file -file=threads.txt

# Heap info
jcmd <pid> GC.heap_info

# Virtual threads
jcmd <pid> Thread.vthread_summary

# GC stats
jstat -gcutil <pid> 1000 10

# Heap dump
jmap -dump:format=b,file=heap.hprof <pid>

# JFR recording
jcmd <pid> JFR.dump name=recording filename=recording.jfr
```

### JOTP-Specific Commands

```bash
# Process summary
jcmd <pid> ProcSys.summary

# List processes
jcmd <pid> ProcSys.listProcesses

# Get process info
jcmd <pid> ProcSys.getProcess <procId>

# Set trace level
jcmd <pid> ProcSys.setTraceLevel VERBOSE

# Check version
jcmd <pid> ProcSys.version
```

### Distributed System Commands

```java
// Check cluster status
Set<NodeId> members = distributedNode.getClusterMembers();
NodeId leader = distributedNode.getLeader();

// Check application status
AppStatus status = distributedNode.getAppStatus("app-name");

// Get cluster info
jcmd <pid> DistributedNode.clusterInfo

// Check network connectivity
nc -zv hostname port
```

## Prevention Best Practices

### Development

- [ ] Always use sealed interfaces for messages
- [ ] Enable exhaustive pattern matching
- [ ] Handle exceptions in all handlers
- [ ] Use timeouts on `ask()` calls
- [ ] Enable debug tracing during development
- [ ] Add logging to all handlers
- [ ] Implement proper error handling

### Testing

- [ ] Use Awaitility for async assertions
- [ ] Clean up processes in `@AfterEach`
- [ ] Run tests multiple times to detect flakiness
- [ ] Test with realistic load
- [ ] Verify process cleanup
- [ ] Test failure scenarios
- [ ] Add integration tests for patterns

### Production

- [ ] Set up monitoring and alerting
- [ ] Implement health checks
- [ ] Use bounded mailboxes
- [ ] Enable distributed tracing
- [ ] Regular heap dump analysis in staging
- [ ] Monitor GC pause times
- [ ] Profile before deploying
- [ ] Test failure scenarios
- [ ] Document runbooks
- [ ] Practice incident response

## Escalation Guidelines

### Level 1: Self-Service

**Time to Resolution:** 1-2 hours

- Use troubleshooting guides
- Follow diagnostic workflow
- Try documented solutions
- Search existing issues

### Level 2: Community Support

**Time to Resolution:** 1-2 days

- Post on GitHub Discussions
- Include issue report template
- Share diagnostic information
- Engage with community

### Level 3: Core Team

**Time to Resolution:** 2-5 business days

- Create GitHub issue with full details
- Include all diagnostics
- Provide minimal reproducer
- Be available for follow-up questions

### Level 4: Emergency

**Time to Resolution:** < 4 hours (production down)

- Contact via private support channel
- Subject: "URGENT: Production Issue"
- Include:
  - Impact assessment (users affected, revenue impact)
  - Current workaround (if any)
  - All diagnostic information
  - Timeline of events
